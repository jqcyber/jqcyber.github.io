Overview

T-Pot is an all-in-one honeypot platform developed by Deutsche Telekom Security. It bundles over 20 honeypot daemons — including Cowrie, Dionaea, Honeytrap, and more — into a single Docker-based deployment, giving defenders a realistic attack surface to observe adversary behavior in the wild.

The goal of this project was to deploy T-Pot in a controlled environment, expose it to the internet, and collect real attack data from threat actors — covering everything from brute-force attempts and credential stuffing to malware drops and exploitation attempts.

Objectives
  • Deploy and configure T-Pot on a Linux host exposed to the internet
  • Collect live attack telemetry from real-world threat actors
  • Analyze attacker TTPs, source IPs, payloads, and credentials attempted
  • Build familiarity with threat intelligence collection and log analysis
  • Explore the Kibana dashboards built into T-Pot for visual threat analysis
Tools & Environment

T-Pot was deployed across two Azure cloud virtual machines in a Hive-Sensor architecture — separating the management and visualization layer from the honeypot sensor layer for better isolation and scalability.

  • T-Pot — multi-honeypot platform (Docker-based)
  • Microsoft Azure — cloud hosting for both VMs
  • Linux — host OS for both the Hive and Sensor
  • Docker / Docker Compose — container orchestration
  • Kibana — built-in visualization and log analysis dashboard
  • Elasticsearch — log storage and indexing backend
  • Cowrie — SSH/Telnet honeypot for credential capture
  • Dionaea — malware capture honeypot
Deployment Process

Important: T-Pot is intentionally exposed to the internet — deploy it on a dedicated, isolated host and never on a machine with access to sensitive data or production networks. A system compromise can never be fully ruled out. Running T-Pot on a home network can attract malicious traffic that puts the rest of the network at risk — a cloud-based VM is strongly recommended.

T-Pot uses a distributed Hive/Sensor architecture. The Hive is the central management and visualization node — running Elasticsearch, Kibana, Logstash, and the web UI. The Sensor runs the honeypot daemons and ships all captured log data back to the Hive over an encrypted SSH tunnel.

Step 1 — Provision Azure VMs

Two Azure virtual machines were provisioned: the Hive (16 GB RAM, 256 GB SSD) and the Sensor (8 GB RAM, 128 GB SSD), each running a clean Linux install. Both VMs were placed on public subnets with no inbound firewall restrictions on honeypot ports, ensuring real attacker traffic could reach the Sensor unobstructed. Management access was restricted to a non-default SSH port.

Step 2 — Install T-Pot on the Hive

On the Hive VM, the T-Pot installer was pulled and executed as a non-root user. When prompted for installation type, H (Hive) was selected, which installs the full Elastic Stack, NGINX reverse proxy, Kibana dashboards, the GeoIP Attack Map, and all management tooling.

bash — Hive installation 
# Run as a non-root user from $HOME
env bash -c "$(curl -sL https://github.com/telekom-security/tpotce/raw/master/install.sh)"

# Select H for Hive when prompted
# Set a WEB_USER and password for the T-Pot WebUI (BasicAuth)
# Installer will configure Docker, Logstash, Kibana, NGINX, and reboot

Once the installer completes, the Hive reboots automatically. T-Pot remaps the system SSH port to 64295 to free up port 22 for the Cowrie SSH honeypot. After reboot, the Kibana dashboard and Attack Map are accessible at https://<hive-ip>:64297.

Step 3 — Install T-Pot on the Sensor

On the Sensor VM, the same installer was run, this time selecting S (Sensor). The Sensor installs only the honeypot daemons and the Logstash forwarder — no Kibana or Elastic Stack. The Sensor does not become active until it is linked to the Hive in the next step.

bash — Sensor installation 
# Run on the Sensor VM as a non-root user from $HOME
env bash -c "$(curl -sL https://github.com/telekom-security/tpotce/raw/master/install.sh)"

# Select S for Sensor when prompted
# Reboot once installation completes

Step 4 — Connect the Sensor to the Hive (Distributed Deployment)

After both VMs are up, the Sensor must be linked to the Hive so it can securely forward its honeypot logs. This is done by running the deploy.sh script from the Hive, which generates SSH keys and configures an encrypted Logstash tunnel between the two nodes.

bash — linking Sensor to Hive 
# On the Hive, generate an SSH key for the Sensor connection
ssh-keygen  # leave passphrase empty

# Deploy the Sensor from the Hive
cd ~/tpotce
./deploy.sh  # follow prompts: enter Sensor IP and credentials

# Verify all containers are running on both nodes
dps.sh

Once linked, the Sensor begins transmitting honeypot event data to the Hive's Elasticsearch instance over a hardened SSH tunnel on port 64295. The Hive user created for this connection is restricted to log forwarding only — it cannot open an interactive shell.

Step 5 — Verify and Monitor

With both nodes running, honeypot data flows into Kibana in real time. The T-Pot landing page at https://<hive-ip>:64297 provides access to the Kibana dashboard, GeoIP Attack Map, CyberChef, Elasticvue, and SpiderFoot — all pre-configured and ready to use.

Results & Findings

Shortly after going live, the honeypot began receiving a significant volume of inbound attacks. Analysis of the initial data revealed over 5,000 attacks within the first 24 hours across the honeypot services, with the vast majority targeting Cowrie, the SSH honeypot — confirming that SSH remains one of the most actively scanned and attacked services on the internet.

Further inspection of the SSH brute-force data revealed consistent patterns in attacker behavior. The most commonly used username was root, and the most common password attempted was 12456 — highlighting how attackers rely heavily on default and weak credentials when targeting internet-exposed SSH services.

Notable: 99% of source IPs were already flagged as known malicious actors in threat intelligence databases — confirming that most of this activity comes from established botnets and repeat offenders.

The Kibana dashboard provided a real-time visual breakdown of attack telemetry — including origin countries, most targeted ports, top attacker IPs, and credential attempts — turning raw log data into immediately actionable threat intelligence.

Key finding: The concentration of attacks on Cowrie and the dominance of weak credential attempts confirm that SSH is one of the most targeted services on the internet — and that most attackers rely on automation and known-bad credential lists rather than sophisticated techniques.

Lessons Learned
  • Exposure is instant — internet-facing systems are discovered and probed almost immediately after going live
  • Weak credentials are the primary attack vector — root/12456 being the top combo shows attackers default to trivially weak credentials first
  • Known threats dominate — 99% of source IPs were already in threat intel databases, reinforcing the value of IP reputation feeds
  • SSH is a high-value target — the overwhelming focus on Cowrie underscores how critical SSH hardening is for any internet-exposed host
  • Cloud-based honeypots are practical — Azure made it straightforward to spin up isolated, dedicated VMs without impacting other infrastructure
  • Visualization accelerates analysis — Kibana dashboards turned thousands of raw log lines into immediately readable attack narratives
